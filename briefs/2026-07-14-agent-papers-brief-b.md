## 1. VIA：把机器人控制重塑为「视觉界面上的智能体任务」

- **arXiv**: 2607.11119 ｜ **提交**: 2026-07-13 ｜ **机构**: 斯坦福大学
- **作者**: Hengyuan Hu, Priya Sundaresan, Jensen Gao, Dorsa Sadigh
- **主类**: cs.RO / cs.AI

### 摘要
机器人操作需要视觉理解、物理推理、规划与闭环控制。主流路线是把视觉-语言模型微调成 VLA（vision-language-action）模型直接输出低层动作，但受限于机器人数据与算力，VLA 往往比前沿基础模型（FM）小几个数量级，因而缺乏物理推理与长程规划能力。VIA 反其道而行：**不做任何机器人专用微调**，让现成 FM 智能体通过一个基于浏览器的 3D 界面操控机械臂——截图观察、下达直觉指令（摆放虚拟目标夹爪作为路点）、观察结果、再调整。用 Claude Code 与 Codex 两个「编码智能体」零样本求解桌面操作任务；最强模型（Fable 5）在三个 LIBERO-Goal 任务上 96.7%、七块积木排彩虹的长程任务上 100%。

### 方法
- **核心思想**: 把机器人控制**重铸为视觉工具使用任务**（agentic task），与「电脑使用」同构。智能体只感知界面截图，无特权状态访问。
- **浏览器 3D 界面**: 用 RGB-D 相机重建的 3D 点云场景 + 第三人称/腕部相机原始视频流；借鉴遥操作界面。
- **MCP 工具集**（人类可读、通用、少量）: `screenshot`、`gripper_teleport_via_click(u,v)`、`gripper_rotate(axis,deg)`、`gripper_show_rotation_gizmo`、`execute_waypoint`、相机视角工具、`end_episode`。智能体像在 Blender 里摆物体一样先摆「虚拟目标夹爪」作为路点，满意后调 `execute_waypoint` 由简单控制器把真实夹爪移到该位姿。
- **观察-行动闭环**: 观察→规划→执行一个路点→看结果→重规划，直到成功或超时。**无 Code-as-Policies 式代码生成、无感知/技能原语**——感知与推理完全交给智能体本体。可选：以纯文本形式提供一条示例路点列表作为「演示」。

### 相关工作
- **FM 作为机器人策略**: VLA（RT-2、π 系列）微调 FM 输出低层动作，受数据/算力所限规模小、物理推理弱。
- **Code-as-Policies（CaP）**: FM 生成调用感知 API 与人工技能原语的程序；FM 通常不直接感知场景，受制于人工抽象——当原语从高层换成低层时成功率骤降。
- **电脑使用智能体**: 前沿智能体已能通过原生截图/点击/键盘闭环操作真实软件。VIA 正是把这一能力迁移到机器人。

### 实验与结果
- **任务（6 个桌面操作）**: Stack（叠方块）、Turn on stove（转旋钮）、Open drawer、Put bowl on plate（需侧壁捏取+精确居中）、Rainbow（七块排彩虹，长程）、T-block（物理拼搭+精确）。均在 robosuite 仿真；除彩虹外为二值成功检测。
- **主结果（成功率）**: 全部「智能体-模型-提示」配置均能端到端零样本完成，整体成功率从 Codex-5.5 的 60% 到 **CC-Fable 的 88%**。CC-Fable 在 LIBERO-Goal 三任务上 **96.7%**、彩虹 **100%**。
- **文本演示提升**: 给一条示例路点列表，把 CC-Opus 的 LIBERO-Goal 从 77% 提到 **100%**。
- **能力随模型缩放**: 更强的底座模型 → 更高成功率；因 VIA 不改动模型/智能体，感知/推理/上下文管理/工具使用的进步可**直接迁移**到机器人控制。
- **成本**: 多数任务每个成功回合 API 成本约几美元。

### 结论 & So-what
「给对界面 + 通用工具，现成前沿智能体已具备可直接迁移到机器人控制的技能」。**对实践者的意义**: 若你在做机器人操作，除了训 VLA，另一条低门槛路线是**把机器人包成一个视觉界面 + MCP 工具，直接挂现成智能体**——闭环纠错、重规划、长程能力免费继承，且随底座模型升级而自动变强。局限: 目前主要在仿真桌面场景、路点式控制、依赖 3D 界面重建。**属于「可立即试」的新范式，而非纯实验室结论。**

## 2. AgentAbstain：LLM 智能体知道「何时不该动手」吗？

- **arXiv**: 2607.10059 ｜ **提交**: 2026-07-11 ｜ **机构**: 伊利诺伊大学厄巴纳-香槟分校（UIUC）
- **作者**: Xun Liu, Yi Evie Zhang, Vira Kasprova, Parisa Rabbani, Pardis Sadat Zahraei, Tianyu Zhang, Ali Ebrahimpour-Boroojeny, Varun Chandrasekaran
- **代码/数据**: agentabstain.github.io ｜ **主类**: cs.AI

### 摘要
智能体评测大多只看任务成功率，却忽视「在歧义、约束冲突或工具失败时，智能体是否知道该弃权」。这一空白有真实风险：智能体可能执行本不该做的**不可逆动作**。AgentAbstain 是首个系统性的「智能体弃权」评测框架，核心是一个**配对任务基准**：基于「触发何时可观测」的 agent-native 分类学定义 **8 类弃权场景**（覆盖执行前推理与运行时发现），含 **263 对配对任务**、分布在 **42 个可执行 MCP 沙箱环境**中；每对由一个「应执行」任务与一个仅引入单一触发使执行变错的「应弃权」变体组成。评测 17 个前沿模型 × 4 个智能体框架：最佳模型（Gemini 3.1 Pro）**配对准确率仅 59.5%**，13/17 低于 50%。

### 方法
- **配对设计**: 每对 = 应执行 `T⁺` + 应弃权 `T⁻`（`T⁻` 只引入一个使执行错误/有害/缺乏依据的触发）。**由构造保证：任何「一律执行」或「一律拒绝」的常量策略配对准确率都 ≤50%**——必须真正分辨该做哪一侧。同时度量过度弃权与不足弃权。
- **8 类弃权分类学**（按触发何时可观测组织）: 执行前类（仅凭指令+工具描述即可识别，如缺关键参数）与运行时类（需通过工具交互才能发现触发，如运行中出现冲突证据——这一情形在纯文本 QA 弃权中无对应）。
- **可执行环境 + commit 级评测**: 工具分 lookup（只读）/ verify（校验门）/ commit（改状态）。评测 = 对工具调用轨迹的**确定性 commit 检查** + 对终端回复的 LLM 裁判，二者结合以捕捉单一信号看不见的失败模式。
- **AbstainGen（全自动生成流水线）**: 端到端合成沙箱环境与配对任务，用确定性 DAG 重放 + 跨模型族 LLM 批判校验；支持按需再生新实例以对抗数据污染。人工三标注者对 100 对抽样评为 94–98% 有效。

### 相关工作
定位为首个系统性的「智能体弃权」评测；区别于纯文本 QA 弃权（无「运行时发现触发后能否刹车」这一 agentic 维度）。有并行工作（脚注引用）通过改造既有基准研究智能体弃权，本文以原生配对设计与 commit 级评测区分。

### 实验与结果
- **主结果**: 17 个前沿 LLM 配对准确率从 GPT-4o 的 33.0% 到 Gemini 3.1 Pro 的 **59.5%**，均值 45.7%；前二名 Gemini 3.1 Pro（59.5%）与 Claude Opus 4.7（59.4%）几乎并列，差距 <0.2pp；存在 ~60% 的天花板，与厂商/规模/发布日期无关。
- **关键发现①（能力解耦）**: **弃权能力与通用任务求解能力近乎独立**——单纯把「会做事」做强并不能补上弃权缺口。
- **关键发现②（post-hoc 弃权）**: 独有失败模式——智能体**先执行了不可逆动作，事后才「声明」拒绝**（例：先取消航班，才发现无法改签，用户被困）。文中还区分 implicit abstention（不动手但也不告知用户，同样判失败）。
- **类目差异**: 配对准确率跨 8 类相差 2× 以上；部分类目模型靠「弃权侧」得分、部分靠「执行侧」得分，只有提升弃权侧才显著改善配对准确率。

### 结论 & So-what
「会不会做」和「知不知道别做」是两种能力，且后者不会随模型规模自动解决。**对实践者的意义**: 若你的智能体能执行不可逆动作（下单、取消、删除、转账、改配置），**不要用任务成功率来推断它的安全性**——必须单独测「弃权/澄清校准」，尤其防范「先动手、后声明」这一 post-hoc 模式，并在工具层为不可逆动作加硬门（先 verify 再 commit）。基准与生成器已开源，可做回归评测。

## 3. ToolAtlas：把工具知识放到「工具侧」——一次学习、处处复用

- **arXiv**: 2607.11126 ｜ **提交**: 2026-07-13 ｜ **机构**: 北京大学计算机学院 · 微软
- **作者**: Yue Fang, Zhibang Yang, Fangkai Yang, Xiaoting Qin, Liqun Li, Qingwei Lin, Saravan Rajmohan, Dongmei Zhang
- **代码**: https://github.com/PuppyKnightUniversity/ToolAtlas ｜ **主类**: cs.LG

### 摘要
LLM 智能体越来越依赖由共享 provider 提供、被异构下游智能体调用的外部工具（MCP 生态）。现有做法在**智能体侧**改进工具使用（改参数/改 prompt/agent 侧记忆），导致工具知识难以共享、且只覆盖过去任务中观察到的行为。ToolAtlas 主张：可复用的工具知识应由**工具 provider 侧**维护。它为每个工具服务器构建持久的图式工具记忆——记录工具能力、失败边界与跨工具组合，通过「执行验证式探测」构建；推理时智能体经**自适应图遍历**查询。在两个 MCP 基准（跨 8 个服务）上，pass@1 最高 **+21.61%**、pass@4 **+18.61%**；同一记忆还能**跨环境实例与跨智能体框架迁移**（免重训、免任务期探索），带来最高 24.16%/16.22% 的相对增益。

### 方法
- **provider 侧三层记忆图**:
  - **Tool-Trace Graph**: 把过去任务执行蒸馏成「智能体中立」的工具-理由轨迹。
  - **Tool-Capability Graph**: 给每个工具挂 affordance（能做什么）与 boundary（真实边界/失败条件）条目，并用「共用（co-usage）边」连接工具。
  - **Tool-Strategy Graph**: 从大量任务中抽取反复出现的编排原则。
- **Frontier Exploration（前沿探测）**: 用主动探测替代被动轨迹收集——LLM 沿两个互补方向探测图的当前空白（向外探未测的边界条件、向内探未验证的 affordance 与工具组合），对**真实工具**执行每次探测，把验证结果折回图中，逐轮锐化。这样图沿「每个工具的能力前沿」扩展，而非沿偶然的任务覆盖扩展。
- **推理期自适应遍历**: 轻量 navigator 逐步展开相关区域，而非检索静态 top-k 切片。
- **核心论点**: 因为图与工具一起维护，其构建成本可在众多下游智能体间**摊销**；知识不再绑定于某个智能体的工作流或动作格式。

### 相关工作
对比智能体侧优化（参数更新、prompt 精化、agent 侧记忆）——工具知识难共享、受限于历史任务行为。ToolAtlas 属工具侧优化，但相较「仅改进工具呈现（如描述改写）」的既有工具侧方法，它额外**主动探测能力边界**并组织为可遍历的图记忆。

### 实验与结果
- **基准**: MCPMark（Filesystem、GitHub、Notion、Playwright、PostgreSQL 等）+ MCP-Universe（排除与 MCPMark 重叠域），跨 8 个服务；用 SwitchAct，主模型 GPT-5.4，另测 GPT-5.4-mini。
- **主结果**: 在同环境划分下，ToolAtlas 在多数服务上优于最强基线；相对增益（Impro%）示例可达 **+16.66%**，整体 pass@1 最高 **+21.61%**、pass@4 **+18.61%**。
- **迁移**: 同一工具记忆跨环境实例与跨智能体框架迁移，pass@1/pass@4 分别最高 **+24.16%/+16.22%** 与 **+17.49%/+14.27%**，**无需重训或任务期探索**。
- **消融**: 增益来自「以工具为中心的记忆组织」+「能力导向的执行探测」二者结合。

### 结论 & So-what
「工具侧记忆」是一种有效且可复用的新范式。**对实践者的意义**: 若你运营一个被多个智能体调用的工具/MCP 服务器，与其让每个下游智能体各自试错、各自攒 agent 侧记忆，不如在**服务器侧构建一次能力/边界/组合的图记忆**——成本摊销、跨智能体共享、随工具演进而更新，且下游免重训即可提升调用成功率。**面向工具/平台方的可落地工程范式，代码已开源。**

## 4. Multi-Agent LLMs Fail to Explore Each Other（MACE）：多智能体「互相探索」的失败与修复

- **arXiv**: 2607.11250 ｜ **提交**: 2026-07-13 ｜ **机构**: 威斯康星大学麦迪逊分校 · 加州大学圣塔芭芭拉分校（UCSB）
- **作者**: Hyeong Kyu Choi, Jiatong Li, Wendi Li, Xin Eric Wang, Sharon Li
- **代码**: https://github.com/deeplearning-wisc/mace ｜ **主类**: cs.MA / cs.AI

### 摘要
探索（exploration）是多智能体系统可靠自治的前提，但 LLM 智能体在互相交互时能否有效探索尚不清楚。本文表明：**现代 LLM 智能体做不到**——常表现出短视、极化的交互模式，导致协调次优、遗憾（regret）升高。作者把该问题形式化为「多智能体探索问题」（建模为部分可观测随机博弈 POSG）：智能体必须**试探同伴**以推断其能力并找到有效交互策略。为此提出 MACE（Multi-Agent Contextual Exploration）——一个通过**结构化同伴选择**显式促进探索的轻量框架。在 contextual 与 parametric 两类多样性设定下，MACE 显著改善探索行为与下游任务表现；理论上进一步证明「探索的价值随智能体多样性增大」。

### 方法
- **问题形式化**: 多智能体探索建模为 POSG；每个智能体需在同伴成功率未知、且同伴本身也在随交互演化（非平稳）的情况下，通过探索识别可靠协作者。
- **诊断性实验**: 用一个**双臂老虎机（two-armed bandit）**设定——LLM 智能体反复在两个未知成功率的同伴间选择；发现 GPT-4、GPT-5 等均出现「过早承诺（premature commitment）」，锁定早期偏好而不继续探索。
- **MACE**: 把难解的联合探索问题**分解为各智能体独立的 contextual bandit 序贯决策**；把每次潜在交互的关系结构编码为上下文表示（反映在给定上下文下查询某同伴的效用），据此在「同一同伴不同上下文」间做细粒度探索。用探索奖励项显式激励探索。
- **理论**: MACE 达 O(√(T log T)) 累积遗憾，而贪心非探索策略为 Ω(δT)（δ 为智能体池的能力多样性）——多样性越大，探索价值越高。

### 相关工作
对比多智能体辩论/协作中**固定通信结构**（如各智能体总选右邻）与「仅靠 in-context prompt 让智能体自主平衡探索-利用」的做法。关键发现是：仅靠 prompt 的 in-context 探索**不足以**弥补，必须引入算法结构。

### 实验与结果
- **设定**: ① Contextual diversity——HotpotQA distractor 模式（2 证据 + 8 干扰，分给 10 个 Qwen2.5-7B 智能体，R=5 轮，评 EM/F1，600 难样本）；② Parametric diversity——GPT-5 / Qwen2.5-7B / Llama3.1-8B / Mistral-7B 四异构模型，在 Math500（L4-5）与 GPQA（Diamond）上 R=3 轮。
- **基线**: In-Context Exploration（同等信息、prompt 引导自主选择）、Random、Pre-defined（固定右邻）。
- **核心结果①**: **仅靠 prompt 的 in-context 探索甚至差于随机**——在 contextual diversity 下，其累积遗憾高于 Random 基线；说明「靠提示」系统性地产生糟糕探索。极端例：即便 GPT-5 也会在 297 次交互中 294 次死锁选同一个 Qwen-7B。
- **核心结果②**: MACE 在探索阶段显著降低遗憾，且**增益延续到利用阶段**（冻结所学线性参数后仍优于所有基线）；在 Math500/GPQA 上趋势一致。**连系统中最强的 GPT-5 也从探索机制中受益**——说明探索不足是当前多智能体 LLM 的**根本性**局限，而非弱模型独有。
- **迁移**: 把 HotpotQA 上学到的参数冻结迁移到 2WikiMultihopQA（纯利用模式），仍有效。

### 结论 & So-what
「把多个 LLM 拼成多智能体系统就会涌现好协作」是一种一厢情愿——它们不会可靠地互相探索，且**加提示不解决、有时更糟**。**对实践者的意义**: 构建多智能体/多角色系统时，同伴/协作者的选择不要交给「模型自己看历史决定」，而应加一层轻量的显式探索机制（如 contextual-bandit 式路由）；智能体池越异构，这层机制的收益越大。**属于「诊断出普遍陷阱 + 给出轻量可插拔修复」的信号型工作。**

## 5. AgentFootprint：把「磁盘留存」列为 LLM 智能体的一等评测指标

- **arXiv**: 2607.11149 ｜ **提交**: 2026-07-13 ｜ **机构**: 香港理工大学（工业与系统工程系 / 计算机系 / 生成式 AI 研究院）· 浙江师范大学
- **作者**: Chenglin Yu, Hongquan Gui, Ying Yu, Hongxia Yang, Ming Li
- **主类**: cs.AI

### 摘要
现有智能体基准衡量任务完成、可靠性与推理成本，却**不测智能体一次运行留在磁盘上的持久数据**（日志、上下文快照、检查点、调试轨迹）。AgentFootprint 是（作者所知）首个跨框架的「运行后存储足迹」基准，提供一套**序列化感知**的指标：总留存、通道构成、重复度、增长指数、可压缩性、以及「对话历史可重建性」评分。它揭示一个测量陷阱：朴素的字节级测量会把重复度**低估一个数量级**（数据库分页与 JSON 转义掩盖了重复内容）。结论：同样模型/工具/任务下，**100% 准确率的配置之间留存字节相差 15.7×**；内容寻址存储可减 **4.8–32.7×** 而保留全部可重建性。

### 方法
- **指标套件**（`S_total` 总留存、通道构成、重复度 D、增长指数 α、可压缩性 C、0–3 级历史可重建性 R）: R 询问「能否从留存数据精确重建对话历史」；序列化感知以避免朴素字节测量的低估。
- **固定轨迹对照（fixed-trace control）**: 把**同一条轨迹**在 7 个持久化框架中重放，分离「智能体产生的逻辑数据量」与「持久层放大」——同一轨迹留存量相差 **6.7×**。
- **压力测试**: 重复观测任务上，全历史持久化框架的增长指数 α 高达 **1.95**（近平方级）——200 轮监控一个 2KB 状态文件可留下 **323MB**。
- **公开轨迹审计**: 对 108 份实例归一化的 SWE-bench Verified 提交做导出轨迹体量分析。
- **缓解方案**: 内容寻址存储（content-addressed store）。

### 相关工作
对比现有智能体基准——覆盖任务完成（GAIA、SWE-bench）、可靠性、推理成本，但**无一纳入运行后存储足迹**。本文将「持久存储」提升为应与准确率、可重建性**联合报告**的资源指标。

### 实验与结果
- **发现①（等准确率下的离散）**: 在同为 100% 任务准确率时，留存字节相差 **15.7×**（LangGraph vs. CrewAI）；且有框架什么都不留存 → 无法恢复或续跑。
- **发现②（增长机制）**: 每步持久化全历史的框架 α 高达 1.95（超线性）；带窗口式上下文管理的框架接近线性。
- **发现③（残留主导产物）**: 当智能体产出合法输出文件时，框架残留（日志等）可比这些产物大最高 **24,033×**。
- **公开提交离散**: 108 份 SWE-bench Verified 提交的每实例导出体量跨 **1,617×**（约三个数量级），且**与解决率无可检测相关**——更重的轨迹并不更成功。
- **缓解**: 内容寻址存储去掉大部分留存字节，所有 restore 检查通过、每个 R 分数复现——**精确历史重建并不需要兆字节级留存**。

### 结论 & So-what
存储之于智能体，如同内存之于程序：是一个必须与准确率、可重建性并列上报的资源维度。**对实践者的意义**: 在桌面部署、数据留存合规、或车队级规模下运行智能体时，别只盯准确率——**同等效果的框架在磁盘留存上可差一到两个数量级**；优先选带窗口式上下文管理的框架，或引入内容寻址存储，可在不牺牲审计/恢复能力的前提下大幅降存储。**面向智能体部署与运维的可量化、可落地指标。**
