## 1. ScaleCUA：以可验证任务合成与高效在线 RL 扩展计算机使用智能体

- **arXiv**: 2607.11185 ｜ **提交**: 2026-07-13 ｜ **机构**: 清华大学、Z.AI
- **作者**: Bowen Lv, Xiao Liu, Yanyu Ren, Hanyu Lai, Bohao Jing, Hanchen Zhang, Yanxiao Zhao, Shuntian Yao, Jie Tang, Yuxiao Dong
- **代码/模型/数据**: https://github.com/THUDM/SCALE-CUA

### 摘要
计算机使用智能体（CUA）通过视觉感知与 GUI 操作自动化数字工作流。可验证奖励的在线强化学习（RLVR）是扩展其能力的关键方向，但受制于**可验证数据稀缺**与**在线 RL 效率低下**两大瓶颈。ScaleCUA 提出统一框架：数据侧用 VeriGen 端到端合成可验证 RL 任务；训练侧用 Frontier Sampling 与 Visual Context Segmentation 提效。在 Qwen3.5-9B 上取得 OSWorld 68.7%、ScienceBoard 54.0%，为开源 CUA 新 SOTA。

### 方法
- **VeriGen（可验证任务合成）**: 三个独立 LLM 智能体——proposer（提任务+判据）、judger（静态语义审查）、checker（在 docker 内「dry run」动态验证目标可达且判据正确）构成多智能体反馈环；共享 docker 交互探针支持 100+ worker 并发。轨迹引导合成：失败任务定位首个偏离步并拆成更易子任务；成功任务聚类后链成更难的多目标任务。共产出 24K+ 可验证任务、近 3K 高质量 RL 任务。
- **Frontier Sampling（高效采样）**: 维护每任务成功率的 EMA，用高斯核（μ=0.5, σ=0.25）把成功率映射为采样权重，把 rollout 集中到「当前能力边界」的任务，另留 γ=0.2 均匀采样保探索。
- **Visual Context Segmentation（长程多模态训练）**: 滑动窗口只保留最近 K 张截图（token-ID 流 + 文本历史流），把训练项从 step-wise 的 O(nT) 降到 O(nT/K)，K=5–8 最优，端到端提速 **2.83×**。
- **训练栈**: 基于 AgentRL，vLLM 做 rollout，Megatron-LM 做训练，算法为 GRPO + DAPO 式非对称裁剪 + KL 惩罚。

### 相关工作
- **CUA 基准**: 从 Mind2Web/WebArena 演进到 OSWorld、WindowsAgentArena、ScienceBoard。
- **GUI 数据生成**: AgentTrek、OS-Genesis、GUI-Genesis、OpenCUA 等——多聚焦推理轨迹或轻量重建环境，缺少在真实桌面 OS 中的**可执行判据合成**。
- **GUI 在线 RL**: DigiRL（安卓）、WebRL（课程式）、ComputerRL、EvoCUA、DART、MobileRL 等——受限于均匀采样浪费 rollout、step-wise 分解拖慢训练。ScaleCUA 针对这两点分别用 Frontier Sampling 与 Visual Context Segmentation 解决。

### 实验与结果
- **OSWorld**: ScaleCUA-Qwen3.5-9B 达 **68.7%**，超越最强开源 Kimi K2.5（63.3%）与闭源 Claude Sonnet 4.5（62.9%）、Seed-1.8（61.9%）；Professional 89.5%、Workflow 48.1% 长程类目提升最大。三个 backbone 均一致提升（Qwen3.5-9B: 41.8%→68.7%, +26.9），证明框架**模型无关**。
- **ScienceBoard**: 54.0%，超 Claude Opus 4.6（52.7%）为新 SOTA；TeXstudio 86.7%、Lean 定理证明 19.0%（所有闭源基线 ≤15.4%）。
- **消融**: 去掉 VeriGen 掉回基座 43.9%；去 Frontier Sampling→63.7%；去 Visual Context Segmentation→62.2%，三者互补。VeriGen 多智能体角色消融：完整流水线判据可执行率 94.5%，去 LLM Judge 掉 32.2pt 最大。
- **VeriGen 质量**: 24K+ 候选、近 3K 通过全部过滤；人类专家审计判据与人工标注一致率 82.5%；轨迹引导任务把 OSWorld 从 64.6% 提到 68.7%。

### 结论 & So-what
可验证奖励 + 面向能力边界的采样 + 有界视觉上下文，是一套**可复用的 CUA 训练配方**。**对实践者的意义**：想训自己的桌面智能体，不必等更大模型——9B 级即可通过「合成可验证任务 + 高效在线 RL」达到甚至超越闭源。局限：每回合截至 50 步、仅 Ubuntu 桌面、仅 8–9B 规模验证。**这不是停留实验室，而是可落地的开源方案（代码/模型/数据全开）。**

## 2. StructAgent：以统一因果结构驾驭长程数字智能体

- **arXiv**: 2607.11388 ｜ **提交**: 2026-07-13 ｜ **机构**: 加州大学圣地亚哥分校、Aether AI Lab
- **作者**: Wenyi Wu, Sibo Zhu, Kun Zhou（通讯）, Aayush Salvi, Zixuan Song, Biwei Huang

### 摘要
长程计算机使用任务中，智能体的上下文会不断累积观察、中间编辑、失败尝试与半成品，导致进展难以解释、验证与恢复。StructAgent 主张把**智能体的状态与工作流**都围绕「统一因果结构」显式组织：用一个紧凑、可验证的统一状态维护任务进展，用结构化工作流通过「验证器背书的状态转移」推进进展。在 OSWorld-Verified 上把 Qwen3.5-9B 从 27.0% 提到 46.9%、Qwen3.5-27B 从 31.6% 提到 62.2%，用 MiniMax-M3 达开源 SOTA **78.9%**，并可迁移到 Minecraft。

### 方法
- **统一状态** `S_t = (S_t^req, S_t^val, S_t^ver)`: 当前需求、有用值（文件路径/URL/选中实体）、验证过的证据；为规划/执行/验证提供共享可读写接口。
- **结构化工作流**: `S_t →(Planner) g_t →(Actor) τ_t →(Verifier) d_t →(Update) S_{t+1}`。核心原则：Planner/Actor 只能「提议」进展，**只有 Verifier 背书的决策才能提交、保留或作废状态**。Verifier 用结构化 + 智能体式探测（编码检查、视觉检查、文件探针）。
- **进阶能力**: 进展检查点与恢复（从 `S^req` 恢复无需回放全历史）；失败恢复（从 `S^ver` 诊断→定向干预：重规划/重试/补检/修复环境/人工接管）；工具支持执行（从记忆归纳可复用过程）。

### 相关工作
面向 GUI/数字智能体的长程执行；对比直接在原始交互历史上操作的常规智能体，以及 leaderboard 单模型（Claude Sonnet 4.6、Kimi K2.6、Qwen3.7-Plus）与框架类方法（Agent S3、OS-Symphony、VLAA-GUI）。核心区别在于把「状态」升级为**类型化、可审计、验证器背书**的一等对象。

### 实验与结果
- **OSWorld-Verified**（总体成功率）: 
  - Qwen3.5-9B: 27.0% → **46.9%**（同组最佳）
  - Qwen3.5-27B: 31.6% → **62.2%**（超 Agent S3 的 60.2%）
  - MiniMax-M3: 单模型基线 → **78.9%**，开源新 SOTA（OS 类目 95.5%）
- **Mind2Web**: 作为 web 泛化协议（judge 打分）另行报告，同样有提升。
- **Minecraft**: 用库存（inventory）验证替换桌面证据源，验证跨域通用性。

### 结论 & So-what
把「什么算进展」从模型自报的 DONE 改为**验证器背书的证据**，是长程智能体可靠性的关键。**对实践者的意义**：与其堆更长上下文，不如给智能体加一层显式的、可验证的状态层——同一框架在不同 backbone、不同环境（桌面/web/Minecraft）都稳定见效。属于**可直接改造现有 agent 栈的工程范式**。

## 3. MM-ToolSandBox：视觉工具调用智能体的统一评测框架

- **arXiv**: 2607.11818 ｜ **提交**: 2026-07-13 ｜ **机构**: Apple
- **作者**: Kaixin Ma*, Di Feng*, Alexander Metz, Jiarui Lu, Eshan Verma, Afshin Dehghan
- **代码/基准**: https://github.com/apple/ml-mmtoolsandbox

### 摘要
现有工具调用基准大多以文本为中心，无法评测「用户发来截图/照片、智能体需据此调用工具」这类真实场景。MM-ToolSandBox 提供有状态执行环境，含 **16 个应用域、511 个工具**，支持多图多轮任务，并覆盖目标变更、纠错、状态突变等真实对话现象。自动化场景生成流水线产出 258 个人工核验场景（1,284 张图）+ 50 个 UI 变体。评测 12 个 SOTA 模型（4B 开源到前沿闭源）发现：**最强模型成功率仍 <50%**；53% 失败源于「图像信息提取错误」而非流程错误。

### 方法
- **框架**: 基于 ToolSandbox 扩展，图像作为一等「视觉工件」存入共享图库、带稳定 ID、可跨轮引用；`search_tool` 元工具按需检索工具 + LRU 驱逐控制活动集。两种执行模式：Code-Execution（沙箱 Python）与 Tool-Use（结构化 API），三层安全护栏。
- **三条正交设计轴**: ① 信息流类型（aggregate/compare/filter/compute/validate/lookup chain/cross-reference 共 7 种）；② 挑战类型（目标变更、纠错、状态突变）；③ 图像到达模式（upfront/progressive/late/mixed）。
- **评测**: State-based 验证（初末世界状态 diff）+ 双 LLM 裁判（Agent Judge 五项全过才算成功 / User Judge 校验用户模拟一致性）；Agent Judge 与人工一致率 88%（κ=0.748）。
- **6 阶段生成流水线**: 图-工具关联→CLIP 聚类+流类型筛选→大纲→场景生成→实体生成→质量过滤；再加 oracle 可解性检查（Claude 4.5 Opus 带真值）+ 人工复核。漏斗：10,466 图 → 258 场景。

### 相关工作
工具调用基准（API-Bank、BFCL、ToolSandbox、AppWorld、Gaia2、τ²-Bench、ASTRA-bench、Toolathlon、MCP-Atlas）多为纯文本、单执行模式、依赖人工构造；视觉智能体基准（VisualWebArena、OSWorld、AndroidWorld、MobileWorld）聚焦像素级 GUI 操作。MM-ToolSandBox 定位互补：感知异构视觉输入（照片/文档/图表/UI 截图）并翻译为**高层 API 调用**。

### 实验与结果
- **主结果（Agent SR）**: Claude 4.5 Opus **48.8%**（最高）、Gemini 3.1 Pro 48.1%、KIMI 2.6 41.5%、GPT-5.4(thinking-high) 41.5%；开源 Qwen 3.5 从 4B 的 11.6% 到 27B 的 34.9%。**无一过 50%**。
- **四点发现**: ① 前沿模型仍吃力；② Agent SR / Entity F1 / Avg.Steps 反映不同侧面；③ 推理（thinking）显著提升（GPT-5.4: 29.1%→41.5%）；④ 规模显著提升任务完成度。
- **场景分析**: upfront 比 progressive 难近 20pt → **多图工作记忆是关键瓶颈**；filter/aggregate 类（多图综合）比 lookup chain（单图检索）难近一倍。
- **失败归因**: 53% 失败源于图像信息提取错误（流程本身正确）。**规划-精度交叉**：小模型败在「决定做什么」，大模型败在「看准所见」。
- **接口消融**: Code vs Tool-Use 无定论、模型相关；工具粒度（511→30）亦模型相关。

### 结论 & So-what
视觉工具调用是一种**独立于视觉识别与文本工具调用的能力**，且当前远未解决。**对实践者的意义**：若产品依赖「用户截图/拍照→触发工具动作」，不要假设前沿多模态模型可靠——**视觉精度（看准）与多图工作记忆是当前首要短板**，且优化方向因模型规模而异（小模型补规划、大模型补感知）。基准与框架已开源，可直接用于回归评测。

## 4. Omni-Decision：面向全模态 QA 的渐进式证据状态智能体系统

- **arXiv**: 2607.11433 ｜ **提交**: 2026-07-13 ｜ **机构**: 中科院神经所、国科大、阿里巴巴通义实验室、清华大学、中科院计算所
- **作者**: Ming Ma, Yi Zhu（通讯）, Yiran Zhong（通讯）, Feida Zhu, Weigao Sun, Junhan Shi, Lingrui Mei, Tianming Yang, Steven Hoi

### 摘要
全模态取证 QA 要求智能体回答证据稀疏分布于视频、音频、图像、网页与计算结果中的问题。现有系统常把证据留在草稿、工具轨迹或自由历史里，难以追踪「已锚定什么、还缺什么、何时足以作答」。Omni-Decision 提出**免训练的证据状态系统**，把全模态 QA 变成「查询范围内的证据闭合」过程：为每个查询维护结构化证据状态（已确认证据、未解冲突、事实/计算依赖、开放需求），并以此条件化规划、取证、验证、修复与终止。OmniGAIA 45.6%、WorldSense 58.3%，分别 +27.3 / +30.2pt。

### 方法
- **查询范围证据状态**: 分离只读上下文 `R_0`（原始查询、资产指针、答案格式、工具可用性、预算）与可变状态 `S_t = {E_t, C_t, F_t, U_t}`（确认证据、冲突、事实/计算完成度、开放需求）。
- **推理循环**: `a_t = decide(R_0, S_t; A(R_0))` 选择工具动作或 finish → 工具返回观察 `o_t` → critic 判定支持/冲突/开放 → **reducer** 用确定性字段更新规则提交 `S_{t+1} = S_t ⊕ ΔS_t`。
- **终止条件**: ready（`U_t=∅ ∧ complete(F_t) ∧ C_t=∅`）返回答案；否则继续或在无推进动作时以 insufficient 停止。
- **关键点**: 把异构工具视为「共享控制接口下的观察源」；只有 reducer 能提交状态，使轨迹可检视、可重放、可消融。

### 相关工作
全模态智能体系统（LongVideoAgent、MMCTAgent、OmniAgent、VideoMind、LongVT、VITED）多以多角色编排、planner-critic 协作定义进展；本文与之互补——不引入新感知模型/工具集/多角色拓扑，而是**规定跨推理决策保存证据的状态对象**。强调 verification 与显式状态是可靠推理的关键。

### 实验与结果
- **配置**: planner/critics/finalizer 用 gpt-5.2；默认感知后端 gemini-3.1-pro。
- **OmniGAIA**（主基准）: Omni-Decision **45.6%**，远超官方 base agent（18.33%）。
- **WorldSense**（迁移评测，多选）: **58.3%**。
- **受控诊断（Table 2）**: 换 Qwen 感知 → 33.33%；换 Qwen planner+感知 → 11.39%；**移除状态**后准确率下降（隔离出证据状态的独立贡献，与后端质量分离）。
- **证据链验证**: 40 例人工子目标标注上做证据链验证。

### 结论 & So-what
在全模态、多步取证中，**显式的「证据状态」控制**比单纯堆上下文窗口或堆帧更有效。**对实践者的意义**：构建多模态深度检索/取证类应用时，与其依赖长历史让模型自行重建进展，不如维护一个查询范围内的结构化证据状态来驱动路由/验证/停止——**免训练、即插即用**，且让轨迹可审计。

## 5. Compile, Then Page：面向流程型 LLM 智能体的可执行 SOP 程序与能力门控运行时

- **arXiv**: 2607.11346 ｜ **提交**: 2026-07-13 ｜ **机构**: 香港理工大学、香港大学、浙江师范大学
- **作者**: Chenglin Yu, Li Yin, Ying Yu, Hongxia Yang, Ming Li（通讯）

### 摘要
企业智能体需遵循长程、含条件分支、安全关键的标准操作流程（SOP）。主流做法把整个 SOP 作为文本塞进 prompt，导致跳步、过早动作、错误拒绝。本文把 SOP **编译成可执行伪代码**，并用「程序引导（PG）栈式运行时」执行：符号栈机做记账并只把「活动帧」分页进上下文，LLM 做语义执行。三臂实验（官方文本 / 编译文本 / 编译程序+运行时）跨 6 个模型表明：编译表示从不显著有害，最多提升 16.0pt；而**运行时收益「按能力门控」——两个强模型跨 7 域受益，弱模型在 Bank 掉 14–26pt**。

### 方法
- **离线编译器（无 LLM，确定性，830 程序约 0.1s）**: 把 SOP 依赖树编译成两层伪代码——每个用户目标一个 process 函数，调用 rule 子程序（把约束措辞与验证配方配对，返回 `{holds, evidence}`）。五机制：有状态子程序、传递式验证-门提升、两种析取调度、帧分解、带证据的返回。
- **在线 PG 运行时（栈式虚拟机）**: 单一记账工具 `pg_program_step`（start/call/return/select_branch/emit_artifact/complete/fail），维护调用栈/游标/返回变量，**只暴露活动帧**（Bank 上约 570 字符 vs 常驻约 10⁴ 字符）。状态存于对话之外，可在中断/压缩后重新披露。
- **软执行（管理注意力，非权限）**: 每个偏离仍可发生但在栈审计轨迹中可见；代价是正确性依赖模型忠实执行本地帧——这正是被度量的「能力门」。

### 相关工作
渐进式披露/上下文工程（MemGPT、Agent Skills、InfiAgent）；控制 LLM 的程序（DSPy、LMQL、SGLang、StateFlow）——程序在模型之外调用模型，本文**反转关系**：程序是模型必须遵循的工件，符号运行时逐帧喂给模型；SOP 智能体与硬约束（受限解码/FSM）——本文软执行居中：偏离可发生但可审计。基准用 SOPBench。

### 实验与结果
- **Bank 三臂（DeepSeek-V4-Flash）**: 官方文本 70.4% → 编译文本 86.4%（+16.0, p<0.001）→ +PG 运行时 **92.8%**（+6.4, p=.021），拒绝正确率 **100%**（86/86）。分解看：执行几乎不变（74.4→71.8→76.9），**收益主要来自正确拒绝**（68.6→93.0→100）。
- **能力门控（6 模型）**: 运行时效应从 +6.4（强）到 −26.4（GPT-4o-mini, p<0.001）；两强模型跨 7 域 pooled 58:19 与 75:31（均 p<.001），两弱模型显著为负（59:237）。
- **全程游标消融**: 把活动帧置顶但保留完整程序，可恢复强模型大部分「拒绝」收益（flat/full-cursor/pg = 88.2/91.0/92.9）；选择性可见性再加小增量。
- **机制归因**: 定向状态重建探针 4 个模型均近满分（0.91–0.99）→ 门控**不是**「不能重建状态」，而是**自发的状态纪律**（弱模型在自身轨迹上给活动帧仅 0.15× 注意力份额，干净重放时升到 0.87）。
- **内容 vs 格式（2×2）**: 强模型二者皆受益；弱模型代码格式反而有害（−12.8）。

### 结论 & So-what
「更多脚手架 = 更好」被证伪。**部署准则**：先编译（官方文本表现差时几乎总有益）；代码语法与活动帧分页**仅在通过「模型级纪律检查」后才启用**；弱模型用强执行框架但**不加**活动帧引导；不可逆动作仍需工具层硬护栏。**对实践者的意义**：给客服/银行/合规类流程智能体做工程时，把 SOP 编译成可执行程序 + 只分页活动帧能显著提升合规拒绝正确率——**但必须先测模型能力，否则弱模型会更糟**。
